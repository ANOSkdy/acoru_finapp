# 実装方針書（finance expansion）

- 作成日: 2026-03-24
- 対象: ANOSkdy/acoru_finapp
- 目的: 現行実装を基点に、財務基盤化へ段階的に拡張する実装順序を定義する。

## 1. 目的とスコープ

### 1.1 目的

- 現行の経費台帳処理を、会計レポートと管理指標を扱える基盤へ拡張する。
- 後戻りを減らすため、依存順にフェーズ分割して実施する。

### 1.2 スコープ

- 対象: データモデル、API、画面、運用設計の拡張方針。
- 非対象: 本書内でのコード実装、migrations 作成、運用体制の最終決定。

追記（2026-03-30）:
- Phase 1（会計基盤整備）は実装済み。
- Phase 2（試算表）は最小スコープで実装済み（migration 003, `GET /api/reports/trial-balance`, `/trial-balance`）。
- Phase 3（PL/BS）は最小スコープで実装済み（migration 004, `GET /api/reports/pl`, `GET /api/reports/bs`, `/pl`, `/bs`）。
- Phase 4 以降（Dashboard, 簡易CF, 複合仕訳移行）は未実装。

## 2. 現状サマリー

現行実装では以下が確認できる。

- 領収書アップロード → `receipt_queue` 登録 → Cron + Gemini 解析 → `expense_ledger` 登録。
- `recordlists` / `recordedit` で台帳 CRUD。
- `receipt_queue` / `cron_locks` は migration で確認済み。

本リポジトリ上で未確認の事項:

- expense_ledger DDL not confirmed
- auth/access control not confirmed
- tests/CI not confirmed

## 3. 推奨実行順序

1. Phase 1: 会計基盤整備
2. Phase 2: 試算表
3. Phase 3: PL / BS（実装済み）
4. Phase 4: Dashboard / KPI / 簡易CF
5. Phase 5: 複合仕訳モデル移行
6. Phase 6: 決算運用 / 予実 / 管理会計

## 4. 6フェーズ計画

### Phase 1: 会計基盤整備

- 目的
  - 勘定科目と会計期間の基礎定義を整備する。
- このフェーズを行う理由
  - 後続の試算表・PL/BS 算出の前提になるため。
- 主な対象
  - 科目体系、期間定義、集計ルールの最小セット。
- 想定成果物
  - 基盤データモデル案、集計定義書、入力ルール。
- 影響範囲
  - DB 設計、ledger API 入力仕様、管理画面要件。
- リスク / 未解決事項
  - 現行 `expense_ledger` の正式制約が未確認。
- このフェーズでやらないこと
  - PL/BS 表示の完成、ダッシュボード実装。

### Phase 2: 試算表（実装済み）

- 目的
  - 期間別の借方・貸方残高を検証可能な形で算出する。
- このフェーズを行う理由
  - 財務三表の前に集計整合を固定する必要があるため。
- 主な対象
  - trial balance 算出 API、期間フィルタ、科目別集計。
- 想定成果物
  - 試算表 API 仕様、検証用サンプル、表示要件。
- 影響範囲
  - 集計 SQL、レスポンス構造、画面表示。
- リスク / 未解決事項
  - 開始残高・繰越ルールの業務定義が未確定。
- このフェーズでやらないこと
  - PL/BS 完成、KPI 可視化。

### Phase 3: PL / BS（実装済み）

- 目的
  - 試算表を基に PL / BS を算出・表示する。
- このフェーズを行う理由
  - 財務把握の中心指標を整備し、次フェーズの KPI に接続するため。
- 主な対象
  - 科目マッピング、PL/BS API、最小表示画面。
- 想定成果物
  - migration 004、PL API、BS API、`/pl` `/bs` 画面。
- 影響範囲
  - API 追加、UI 追加、会計定義ドキュメント。
- リスク / 未解決事項
  - 表示科目マッピング方針の合意が必要。
- このフェーズでやらないこと
  - CF API、高度分析、予実管理。

### Phase 4: Dashboard / KPI / 簡易CF

- 目的
  - 財務状況と業務処理状況を統合可視化する。
- このフェーズを行う理由
  - PL/BS が揃うと KPI の意味づけが安定するため。
- 主な対象
  - ダッシュボード画面、KPI 集約 API、簡易 CF 表示。
- 想定成果物
  - KPI 定義、ダッシュボード情報設計、サマリ API。
- 影響範囲
  - フロント UI、集計 API、キャッシュ戦略。
- リスク / 未解決事項
  - 更新頻度とリアルタイム性要件の調整。
- このフェーズでやらないこと
  - 複合仕訳への構造移行。

### Phase 5: 複合仕訳モデル移行

- 目的
  - 単一行中心モデルから複合仕訳を扱える構造へ移行する。
- このフェーズを行う理由
  - 取引実態に沿った明細管理と拡張性確保のため。
- 主な対象
  - `journals` / `journal_lines` への移行設計。
- 想定成果物
  - 移行計画、互換 API 方針、データ移行手順。
- 影響範囲
  - DB、ledger API、編集 UI、集計ロジック。
- リスク / 未解決事項
  - 既存データ互換と切替時停止時間。
- このフェーズでやらないこと
  - 決算運用のフル機能化。

### Phase 6: 決算運用 / 予実 / 管理会計

- 目的
  - 月次決算運用、予実比較、管理会計の運用機能を整備する。
- このフェーズを行う理由
  - 基盤・帳票・可視化・仕訳モデルの整備後に業務運用へ落とし込むため。
- 主な対象
  - 締め処理、予算データ連携、部門別分析。
- 想定成果物
  - 決算運用フロー、予実レポート、管理会計指標。
- 影響範囲
  - API、UI、運用ドキュメント、権限制御要件。
- リスク / 未解決事項
  - auth/access control 未確認のため運用権限設計が前提不足。
- このフェーズでやらないこと
  - ERP 全面統合や外部 BI 全自動連携。

## 5. 依存マップ

- Phase 1 → Phase 2: 科目・期間定義が無いと試算表は安定しない。
- Phase 2 → Phase 3: 試算表の整合が無いと PL / BS の信頼性を確保できない。
- Phase 3 → Phase 4: 財務指標の基礎が無いと KPI の意味づけが不安定。
- Phase 4 → Phase 5: 可視化要件を確認してから複合仕訳移行範囲を確定する。
- Phase 5 → Phase 6: 運用機能は新仕訳モデル確定後に実装する。

## 6. 先に解消すべきリポジトリギャップ

- `expense_ledger` 正式 DDL の確認を最優先とする。
- auth/access control の現状仕様を確定する。
- tests/CI の有無と最低品質ゲートを確定する。

## 7. 実行原則

- 現行実装では事実と拡張案を混在させない。
- 拡張案として段階リリース前提で差分を最小化する。
- 先にデータ定義を固め、次に API、最後に UI を整備する。
- server-only env を維持し、機密値をクライアントに露出しない。
- 仕様未確認項目は「本リポジトリ上では未確認」と明記して進める。
