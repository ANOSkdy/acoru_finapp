# 実装方針書 / 拡張ロードマップ（2026-03-24）

- 対象: `ANOSkdy/acoru_finapp`
- 文書目的: 現行実装を起点に、財務レポーティングとダッシュボード拡張を段階実装するための方針を定義する。
- 注意: 本書は docs レベルの計画であり、実装変更は含まない。

## 1. 目的とスコープ

### 1.1 目的

- 現行の経費台帳基盤を、会計基盤→財務諸表→ダッシュボードへ段階的に拡張する。
- 実装順序の依存関係を明確化し、後戻りコストを抑える。

### 1.2 スコープ

- 対象: ドメイン/データ/API/UI/運用ドキュメントの拡張方針。
- 非対象: 具体的なコード変更、DDL 確定、移行スクリプト確定、運用体制確定。

## 2. 現状サマリ（現行実装）

- 領収書アップロード（Vercel Blob）→ `receipt_queue` 登録 → Cron 解析（Gemini）→ `expense_ledger` 登録の流れがある。
- 台帳 CRUD API（`/api/ledger`, `/api/ledger/[journalId]`）と画面（一覧/編集）が存在する。
- `receipt_queue` と `cron_locks` は migration で確認できる。
- `expense_ledger` はコード上で参照されるが DDL は本リポジトリ上で未確認（not confirmed in this repository）。
- 認証/認可全体、組織スコープ、CI/テスト方針は本リポジトリ上で未確認（not confirmed in this repository）。

## 3. 推奨フェーズ別ロードマップ（拡張案）

> 以下は拡張案としての提案（proposed / not implemented）。

### Phase 1: 会計基盤 + 勘定科目マスタ + 試算表

- 目標
  - 勘定科目の正規化と会計期間軸を定義し、試算表を安定算出できる基盤を作る。
- この順番で行う理由
  - PL/BS/CF やダッシュボード品質は、科目体系と期間整備が前提となるため。
- 成果物（deliverables）
  - `account_master` 相当の管理方針（分類・表示順・集計属性）
  - 会計期間（`reporting_periods`）の管理方針
  - 試算表 API 仕様（例: `/api/reports/trial-balance`）
- 影響（想定）
  - スキーマ: 科目/期間ドメインの追加設計が必要
  - API: 試算表集計エンドポイント追加設計
  - UI: 試算表表示画面（または既存画面拡張）
  - ドキュメント: 集計定義・締めルール追記
- リスク / オープンクエスチョン
  - `expense_ledger` 実体 DDL 未確認のため、既存列制約との整合確認が先行課題。
- Out of Scope
  - PL/BS/CF の最終帳票表現
  - 高度ダッシュボード

### Phase 2: PL/BS 生成

- 目標
  - 試算表を基礎に、PL/BS の期間別算出を可能にする。
- この順番で行う理由
  - 試算表で集計ロジックを安定化した後の方が、PL/BS へのマッピング検証が容易なため。
- 成果物（deliverables）
  - PL/BS マッピング定義（科目→表示科目）
  - API 仕様（例: `/api/reports/pl`, `/api/reports/bs`）
  - 帳票表示 UI 方針
- 影響（想定）
  - スキーマ: 科目マスタへの表示科目属性追加検討
  - API: 期間比較・累計/当期の返却仕様
  - UI: 期間切替、比較表示
  - ドキュメント: 帳票定義、丸め/端数処理方針
- リスク / オープンクエスチョン
  - 会計ルールの業務合意が未確認（not confirmed in this repository）。
- Out of Scope
  - キャッシュ最適化の最終決定
  - 複合仕訳の本格対応

### Phase 3: Dashboard + 簡易 CF

- 目標
  - PL/BS を土台に、経営モニタリング用のダッシュボードと簡易 CF 表示を提供する。
- この順番で行う理由
  - ダッシュボード指標の信頼性は、試算表/PL/BS の整備完了を前提とするため。
- 成果物（deliverables）
  - `/dashboard` の情報設計
  - KPI カード、推移表示、処理状況ウィジェット仕様
  - API 仕様（例: `/api/dashboard/summary`, `/api/reports/cf`）
- 影響（想定）
  - スキーマ: 大きな追加は最小化、必要時のみ集計補助情報を検討
  - API: 複数指標の集約レスポンス
  - UI: カード/チャート/ステータス可視化
  - ドキュメント: KPI 定義、指標更新タイミング
- リスク / オープンクエスチョン
  - 指標の更新頻度と最新性要件の合意が未確認（not confirmed in this repository）。
- Out of Scope
  - 高度な予測分析
  - BI ツール連携

### Phase 4: 複合仕訳 / 予実 / 部門別管理

- 目標
  - 複合仕訳、予算実績比較、部門/プロジェクト軸を扱える構造へ進化させる。
- この順番で行う理由
  - 高度機能は基盤レポート安定後に導入した方が移行影響を制御しやすいため。
- 成果物（deliverables）
  - `journal` / `journal_lines` への進化方針（概念設計）
  - 予実管理・部門軸のドメイン設計
  - 管理ワークフロー（レビュー/承認）設計方針
- 影響（想定）
  - スキーマ: 仕訳モデル再編の可能性が高い
  - API: 登録/集計 API の再設計
  - UI: 複数明細入力、予実/部門別ビュー
  - ドキュメント: 移行計画、互換ポリシー
- リスク / オープンクエスチョン
  - 既存データ移行方式、互換期間の取り扱いは未確定。
- Out of Scope
  - 全社 ERP 統合
  - 権限モデル最終確定

## 4. 推奨実装順序

1. Phase 1（科目/期間/試算表）
2. Phase 2（PL/BS）
3. Phase 3（ダッシュボード/簡易 CF）
4. Phase 4（複合仕訳/予実/部門別管理）

理由: 集計基盤 → 帳票 → 可視化 → 高度運用機能の順で依存が明確なため。

## 5. 依存マップ

- 試算表（Phase 1）完了前に PL/BS（Phase 2）品質を担保することは難しい。
- PL/BS（Phase 2）完了前にダッシュボード（Phase 3）を高度化すると、KPI 定義が不安定になりやすい。
- 複合仕訳/予実/部門別（Phase 4）は、前フェーズの集計定義確定後に着手する方が安全。

## 7. 方針上の留意点

- 現行実装では単一台帳中心のため、拡張時は「互換性維持期間」を明示する。
- 拡張案として、まずドメイン定義（科目/期間/表示科目）をドキュメントで固定し、実装は段階投入する。
- 本書は実装指示書ではなく、設計判断の順序を揃えるための方針書である。
